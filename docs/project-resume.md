# MedAgent — 项目简历

- 项目介绍：面向医疗场景设计的 **ReAct Agent** 系统 — 以统一 ReAct 推理循环为核心 + 多引擎 RAG + 分层记忆 + Harness 容错编排。所有查询统一进入 ReAct 循环，LLM 自主决定调用工具还是直接回答，RAG 检索作为 `retrieve_knowledge` 工具在循环内调用。覆盖疾病咨询、用药指导、科室导诊、检查指标查询等功能，支持多 LLM 提供商动态切换与 SSE 流式输出。

- 技术栈：Python 3.10+、FastAPI、Vue 3、PostgreSQL、Milvus、Elasticsearch、Neo4j、SentenceTransformers、Cross-Encoder、DeepSeek API、ZhipuAI、Qwen、Ollama、SSE、psycopg2、Pydantic

- 核心技术实现：
  - **统一 ReAct Agent 架构**：传统医疗问答系统采用 Router 硬选 RAG/Chat/ReAct/Tool 四种模式的架构，模式之间无法切换，RAG 和 ReAct 各自独立。重构为以 ReAct 循环为核心的单体 Agent 架构：Router 仅返回元数据（query_type/数据源选择），不再决定执行模式；所有查询统一进入 ReAct 推理循环，LLM 在每一步自主决策是调用工具还是给出最终答案。RAG 检索被包装为 `retrieve_knowledge` ReAct 工具，原生工具（剂量计算/科室导诊/正常值查询）通过适配器注册到同一引擎。HarnessOrchestrator 提供 5 阶段状态机（RiskDetect → Route → ReactLoop → SafetyCheck → Complete）的容错编排，每阶段独立异常捕获，强制保证输入安全检测和输出安全审查。
  - **Harness 容错执行引擎**：医疗问答中任意组件（检索/LLM/知识图谱）的瞬时故障不应导致整个对话失败，且用户需要知道当前回答的置信度和证据来源。设计并实现五阶段状态机（Idle → RiskDetect → Route → ReactLoop → SafetyCheck → Complete）的容错编排层：MedPhase 定义阶段枚举，MedStateMachine 强制执行流程约束（ReactLoop 必须经过 RiskDetect、Complete 必须经过 SafetyCheck），并提供 force_transition 用于测试恢复。MedToolWrapper 对每个工具调用封装超时控制（per-tool 可配置，如 kg_search 15s/llm_generate 60s）、指数退避重试（retrieval 类最大 2 次、LLM 类最大 1 次、本地工具不重试）和渐进式降级策略（失败时返回中文友好提示"该服务暂时不可用"而非抛异常）。同时提供 SourceTrace 证据溯源（来源类型/原始内容片段/RRF 分数/排序）和 Confidence 置信度跟踪（high/medium/low/none），以 to_dict 序列化供前端展示。配套四个工厂函数（create_retrieval_wrapper/create_llm_wrapper/create_safety_wrapper/create_builtin_wrapper）将现有 HybridRetriever/SafetyGuard/AnswerGenerator 等组件一步包装为容错执行单元。
  - **混合检索 RAG 架构**：单一向量检索对中文医学术语缺乏精确匹配能力，纯关键词检索则丢失语义泛化，且医学领域依赖的结构化知识（疾病-症状-药品关系）无法被非结构化检索覆盖。基于 Neo4j 知识图谱、Milvus 向量检索（BGE-small-zh-v1.5 Embedding）和 Elasticsearch BM25 关键词检索构建三引擎并行检索层：KG 结构化结果经 NER 实体提取后独立传递，保留关系语义不参与融合；Milvus 与 ES 对同一语料库分别执行 Dense 语义召回和 Sparse 关键词召回，两路结果通过 RRF 倒数排名融合（c=60）实现同文档跨源分数累加，再经 Cross-Encoder 对融合后的候选集二次精排，依据语义匹配度重新排序至 top_k=5。三引擎互补覆盖了结构化关系、语义相似和精确匹配三种检索维度，解决了单一引擎的召回盲区问题。
  - **分层记忆系统**：针对医疗问诊多轮长对话中患者跨会话重复描述病史的痛点，设计并实现 Agent 分层记忆系统，构建短期会话记忆、长期语义记忆与用户健康画像三层架构，基于 Neo4j 构建 Memory Graph，通过 FOLLOWS 时序链与 SIMILAR_TO 语义边维护记忆演化网络，并设计内容自动分类、Hash / Embedding 双阶段去重、Graph-Aware Consolidation、TTL + Importance 动态生命周期管理机制，LLM 异步提取过敏史/慢性病史/用药/手术史等结构化健康信息并以医学正则兜底，基于优先级槽位与 Token 预算裁剪实现 Runtime Context Assembly 动态上下文拼装，提升 Agent 长上下文记忆能力与跨对话用户认知延续。
  - **分级安全防护与检索质量双保险**：医疗场景中 LLM 可能对高风险症状（胸痛、呼吸困难等）给出非紧急回复，或在无检索依据时仍凭空生成医学建议，存在严重的误导致命风险。基于 12 类红色急诊关键词和 13 类黄色就医提示关键词实现两级风险扫描，红色信号在回答开头注入 120 急诊警告，黄色信号在末尾追加 48 小时内就医提醒；同时按检索置信度（high/low/none）三级追加免责声明，并在无检索结果时采用 Tier 2 策略——统一回复"建议咨询专业医生"，杜绝 LLM 凭空捏造医学内容。从风险识别和内容管控两个维度降低医疗法律风险。
  - **PostgreSQL 统一持久化与多 LLM 容灾**：早期基于 JSON 文件的多用户会话和记忆存储存在并发写冲突、数据隔离缺失和不可横向扩展的问题。基于 psycopg2 ThreadedConnectionPool（min 1/max 10 连接池）和四表设计（long_term_memory、chat_sessions、session_messages、user_preferences）实现会话/LTM 记忆/用户偏好的统一持久化，全表按 username 多租户隔离，并支持 LTM embedding float[] 数组列直接入库。配合 AnswerGenerator 的 MD5 LRU 内存缓存（可配容量与 TTL）和主 LLM→Ollama 本地模型自动 fallback 机制，实现存储可靠性和生成可用性的双重保障。
